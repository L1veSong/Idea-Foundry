# 迭代精进方法论

从 v1 到 v8.1 的完整演化路径揭示了一套可复用的工作流迭代方法论。

## 演化弧线

```
v1  五步固定 → 能跑
v2  +TDD+验证 → 有质量
v3  +12阶段+冲突管理 → 结构化
v4  +自适应矩阵+回退+强度 → 灵活
v5  +领域路由+子流程+自进化 → 智能
v6  +标签池+多领域组合+权重打分 → 组合
v7  +能力抽象+动态扫描+加权仲裁+降级兜底 → 自组装
v8  +策略选择器+环境感知+主动适配 → 通用开源级
v8.1 +冲突锁+自定义预设+质量预检+Skill推荐+UI自适应+跨平台 → 产品级
```

## 核心原则

### 1. 从配置到智能
每一版都在减少硬编码。v4 的静态枚举 → v5 的领域路由 → v6 的标签池 → v7 的能力抽象。每次都是「把写死的规则替换为运行时发现」。

### 2. 从不崩到优雅降级
v3 缺Skill=崩 → v7 降级链 → v7.1 人工引导兜底 → v8 主动策略适配。每次都是「把失败模式变成降级路径」。

### 3. 从单人到通用
v5 固化领域 → v7 跨环境自适应 → v8 策略选择器。每次都是「把个人偏好变成可配置选项」。

### 4. 从执行到可观测
v7 日志 → v8.1 质量预检+Skill推荐。每次都是「把黑盒变成白盒」。

## 触发时机

什么时候该迭代工作流:
- 用户说「如果遇到X怎么办」→ 缺容降级
- 用户说「这里能不能更灵活」→ 配置化
- 用户说「别人没这个Skill会不会崩」→ 通用化
- 用户说「我怎么知道里面发生了什么」→ 可观测
- 用户说「这个和那个能不能一起用」→ 组合化
- 用户说「能不能帮我选最优的」→ 策略化

## 反模式

- ❌ 一上来就做最复杂的 — v1 从5步开始, 不是从12步
- ❌ 在用户没抱怨的地方过度设计 — 每次迭代都是用户驱动的
- ❌ 为了通用牺牲核心体验 — 降级要优雅, 但主路径必须极致
- ❌ 把环境问题固化成规则 — 缺Skill是降级, 不是「这个功能不可用」
