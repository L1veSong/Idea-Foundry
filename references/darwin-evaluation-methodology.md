# Darwin 评估方法论

## 核心教训

2026-05-19：Darwin 对 idea-foundry v8.3.2 进行优化，结果**负优化**（砍掉 10 项功能，新增 1 项）。

### 为什么误判为「正优化」

最初对比的是两个**都已损坏**的副本（均被行号前缀污染），而非原始干净版。等拿到真正原始的 v8.3.2 后，真相才显现。

### 正确流程

1. **必须先获取干净的原始版本作为基线**（ZIP、备份、Git tag）
2. 运行 Darwin 优化
3. 对比「原始基线」vs「Darwin 输出」，**不对比两个损坏版**
4. 功能关键词逐项核对（不是只比行数和体积）
5. 判定标准：功能丢失 = 负优化，即使结构/表达改进

### Darwin 的约束

Darwin 文档声明「不改变skill的核心功能和用途 — 只优化怎么写和怎么执行」。实际执行中可能越界。建议在运行 Darwin 时明确给出**保护指令**：

```
以下功能段必须保留，不可删除或合并：
- [列出所有关键功能关键词]
只优化：结构顺序、表达清晰度、格式规范
```

### 提取而非全面采纳

如果 Darwin 部分改进有价值但整体负优化，推荐做法：
- 回归原始版本
- 手工提取 Darwin 唯一有价值的产出（如 idea-foundry 的「检查点铁则」表格）
- 嵌入原始版本
- 不整体采用 Darwin 输出
